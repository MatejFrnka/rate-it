About this app:

  • The app has been created by students of Greenfox Academy
  • It went from an idea to a finished product in 7 weeks
  • We followed agile methodology with week long sprints. (daily scrum, sprint planning, backlog refinement, sprint retrospective)
  • Technologies used:
    • Java Spring
    • Javascript
    • Thymeleaf
    • Cockroach DB
    • Flyway
    • Docker
    • Github Actions for CI/CD
    • Google cloud storage
    • OpenStreetMaps
  • Future plans:
    • Mobile and Web app in React Native

Things we are proud of:

Comprehensive role management and security:

Each user has different permissions for every interest (creator, voter, visitor). In private interests, users need to request for permission to vote and add places. The creator is able to manage users in the interest settings.

We also implemented a system for users to be able to reset their password.

OpenStreetMap integration:

When a user is creating a new place, they can select the location in three ways: 1) based on their current location 2) based on the address (which is translated to coordinates using OpenStreetMaps) 3) by placing a pin on the map

When browsing places or interests, users can filter by the nearest place/interest which allows them to discover new places nearby. Users are also able to see all places together in a map.

Images

We store all of our images on Google Cloud. It ensures high reliability and allows us to scale our storage up as much as we need.

We compress our images down to save on the storage cost and to decrease the overall size of our website - thus decreasing the loading time.

Database

We use CockroachDB as it offers high reliability and automatically creates a backup of our data every hour.

For local development, we use docker to run CockroachDB locally.

For smooth deployment to production, we version our database using Flyway.

Email

We used a third-party email service to send notifications of invitations to a new interest and password resets.

Development

We worked hard to keep the code quality high. We wrote unit and integration tests, followed Google code style and used Github actions. This, together with requiring review from three other team members before merging, ensured we only merged high-quality code.

We also frequently went back and refactored our old code.
